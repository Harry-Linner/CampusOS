## 使用边界

Celechron 1.3.0 是 CampusOS 设计校内数据接入时必须研究的成熟实现。这里的“严格参考”是指复用经过验证的行为模型、失败分类、状态机、缓存策略、诊断能力和测试场景，不是直接复制代码。

Celechron 使用 GPL-3.0。CampusOS 在没有完成许可证兼容性评审和明确开源边界之前，不得复制、改写或移植 Celechron 源码。若未来确需代码级复用，必须先新增 ADR，记录许可证判断、派生作品边界、发布义务和第三方声明。

## CampusOS 强制工程要求

### 1. 认证是可验证的状态机

- 统一身份认证成功不能只以“提交账号密码未报错”为准。
- 必须显式处理 CAS service、一次性 ticket、ticket 回调、各业务系统 Cookie/Session 和最终业务身份确认。
- 每个业务服务必须验证自己是否为非匿名、已授权状态；“有 Cookie”不等于“已登录”。
- 登录态失效时允许单次受控重认证，并避免多个并发请求重复登录或覆盖有效 Session。
- 不同账号、不同服务、不同 HTTP 客户端的会话不能无边界共享。

### 2. 刷新必须支持局部成功

- 课表、成绩、考试、作业、实践、校历等数据源分别返回结果和来源状态。
- 单个数据源超时、认证失效、结构变化或单条脏数据不能拖垮整个刷新任务。
- 刷新结果至少区分：实时成功、缓存成功、安全默认、当前不可用。
- 刷新失败时保留上次有效数据，禁止用空响应、异常响应或未经校验的数据覆盖好缓存。

### 3. 网络请求必须具备真实韧性

- 每个请求必须有连接、响应和正文读取超时。
- 只对明确可重试的网络错误、超时、Session 失效和临时服务错误重试；认证拒绝和数据格式错误不得无限重试。
- 重试次数、退避策略、重新认证次数和最终异常分类必须可诊断。
- 请求取消、应用退出和后台任务结束时必须正确释放资源。

### 4. 刷新并发必须统一协调

- 启动刷新、手动刷新、定时刷新和后台刷新必须进入同一个协调器。
- 同一账号的并发刷新使用 single-flight 或互斥机制：可共享同一任务的调用共享结果，后台任务在前台刷新占用时应让步。
- 禁止重复请求、旧任务覆盖新状态、跨进程/线程并发写缓存和“实际共享任务却显示已跳过”等状态错乱。

### 5. 学年范围不能由固定月份决定

- 正常获取当前及历史学年，同时主动探测下一学年。
- 下一学年未开放、空数据或预期 404 应静默跳过，不影响已有学年。
- 探测出现非预期认证、网络或解析失败时必须记录诊断，但不能污染其他学年结果。
- 探测上界必须考虑入学年份、培养阶段和毕业年份，禁止无界请求。

### 6. 缓存必须携带来源与新鲜度

- 缓存按账号、数据源、学年/学期和接口版本隔离。
- 每份展示数据都能说明来源、写入时间和最近一次实时刷新错误。
- 回退顺序必须明确且可测试，例如：实时素拓明细 → 账号隔离的素拓缓存 → 教务实践分汇总 → 保留已有数据并标记不可用。
- 校历采用在线配置 → 同学期缓存 → 基于最后有效配置生成并校验的安全默认值。

### 7. 解析器以“单条隔离”为默认

- 兼容接口中数字/字符串混用、空值、缺失字段和有限的结构漂移。
- 每条课程、成绩、考试、作业或实践项目单独校验；无法识别的单条记录被跳过并记日志，其余有效记录继续入库。
- 缓存读取必须执行与网络响应同等级别的 schema 校验。
- 解析器输出正式领域类型，不把原始动态 JSON 直接传到视图层。

### 8. 诊断能力是接入的一部分

- 真实数据源上线前必须具备“诊断与测试”入口，而不是只有一句“刷新失败”。
- 日志记录刷新 ID、来源、模块、操作、耗时、重试、重新登录、缓存使用、HTTP 状态和异常类型。
- 导出前必须脱敏账号、密码、Cookie、Session、ticket、token、授权码和带敏感参数的完整 URL。
- 日志必须有容量与文件数量上限，并支持查看、复制、导出和清空。
- 诊断日志不得成为新的凭据泄漏渠道；任何新增字段默认按敏感信息处理，确认安全后才允许输出。

### 9. Adapter 边界必须真实

- mock 只允许替代外部响应数据；认证状态机、HTTP 客户端、重试、缓存、刷新协调、持久化、错误传播和诊断链路不能 mock 成功。
- 每个校内网站使用独立 adapter，共享基础认证、请求、日志和缓存能力，但不能把某一网站的 Cookie、字段或错误规则硬编码进全局业务层。
- 视图只消费统一领域快照和显式数据来源状态，不直接了解 CAS、HTML、Cookie 或某网站的 JSON 结构。

## CampusOS 当前实现状态（2026-07-19）

统一身份认证核心链路已经按本基线实现：同会话 `execution`/Cookie、动态 RSA 公钥、有效 SSO Cookie、本科教务网 Session、素拓正式 `SESSION`、非匿名 `ctx`、账号匹配 `getMyInfo` 最小汇总、请求超时与取消、并发连接互斥、结构化脱敏错误，以及认证成功后才执行的 `safeStorage` 原子持久化。旧版未验证凭据不会被误判为已连接。实现说明见 [ZJU 统一身份认证核心登录](../architecture/zju-unified-auth.md)。

本科与研究生课表、考试、成绩及学在浙大作业已经通过核心托管的业务会话、固定操作、单条解析隔离、账号/provider 隔离 capability 缓存、刷新协调和来源状态进入正式代码链路；考试、DDL 和成绩再由独立 feature 消费，renderer 不直接解析网站响应。研究生实现按 `GrsNew` 行为使用 `https://yjsy.zju.edu.cn/` CAS service、固定 `validateLogin` ticket 交换与主进程内存 `X-Access-Token`，缺失考试时间不会套用 Celechron 旧实现中的兜底钟点。首次连接已按用户明确选择的培养层次执行：研究生只有在固定成绩接口返回有效认证后结构时才原子保存，IPC 不返回 token 或成绩正文。诊断页面已连接真实刷新日志并支持脱敏导出。研究生当前仅通过外部 HTTP fixture，尚未执行真实账号现场测试；实践项目明细、完整多级重认证/重试阶段记录、可信节次与课程日期展开、成绩完整口径和多设备验收仍未完成，因此不能把整个数据接入面宣称为生产就绪。插件拆分与依赖关系见 [Celechron 启发的官方插件集设计](../design/celechron-inspired-plugin-suite.md)。

## 素质拓展平台认证链

此链路来自用户提供的维护者说明，并已与 Celechron 1.3.0 的 `lib/http/zjuServices/sztz.dart` 结构交叉核对：

1. 请求素拓专用 CAS 登录地址：`GET https://zjuam.zju.edu.cn/cas/login?service=https%3A%2F%2Fsztz.zju.edu.cn%2Fdekt%2F`。
2. 使用现有统一认证登录态取得一次性 ticket；CAS 通过 `302 Location` 返回 `https://sztz.zju.edu.cn/dekt/?ticket=<一次性票据>`。
3. 访问 ticket 回调地址，消费一次性 ticket。
4. 素拓验证 ticket 后通过 `Set-Cookie` 签发业务 `SESSION`；不得记录 Cookie value。
5. 携带该 `SESSION` 调用 `POST https://sztz.zju.edu.cn/dekt/ctx`。
6. 解析并确认 `ctx` 表示非匿名有效用户后，调用 `GET https://sztz.zju.edu.cn/dekt/student/home/getSqjl`。

顺序不可交换：只有访问 ticket 回调并取得正式业务 SESSION，再通过 `ctx` 确认非匿名身份，才能认为素拓登录完成。

## CampusOS 上线前验收门槛

任一真实校内 adapter 标记为 `ready` 前，至少覆盖以下自动化测试：

- 登录表面成功但业务身份仍匿名时必须失败。
- SSO/业务 Session 过期后只进行受控重认证，并能恢复请求。
- 网络超时和临时错误按分类重试，超过上限后返回明确错误。
- 一个数据源失败时，其他数据源仍能成功并持久化。
- 前台与后台同时刷新时只执行一次有效任务，不发生旧状态覆盖。
- 下一学年已开放、未开放、空响应、预期 404 和非预期失败分别处理。
- 单条 malformed 数据被隔离，其余记录继续解析。
- 无效或空网络响应不能覆盖最后有效缓存。
- 缓存、默认值和实时数据在领域快照中具有明确 provenance。
- 诊断导出不包含账号、密码、Cookie、Session、ticket、token 或敏感 URL 参数。
- 关键认证、解析、缓存和刷新协调测试使用真实实现，只 mock 外部 HTTP 响应。

### 【开源更新】Celechron 社区维护版：修复课表刷新，新增实践分明细（二三四课分）和诊断日志

大家好，由于lz的安卓端（准确的来说是鸿蒙端）总是出现信息更新失败问题（相信很多uu也遇到这个问题，因为这主要是代码问题不是网页问题），最近我在原版 Celechron 的基础上做了一轮兼容性维护和功能补充（如果原作者想要在此基础上更改，那可能要费一番功夫，因为改了不少原先的代码，但我个人觉得其实很多功能已经比较完善了没有必要再改）。

以下是软件介绍：

Celechron 是一款面向浙大学生的开源时间管理工具，可以查看课表、成绩、考试、作业和日程等信息。由于最近部分校内系统的登录方式、接口返回内容以及新学年数据发生了变化，原版在一些情况下会出现登录成功但刷新失败、数据缺失或者无法读取新课表的问题。

本次更新尽量保留原版界面和使用方式，主要修改数据获取、登录和容错逻辑，没有对无关功能进行大规模重构。

需要说明：这是基于原项目制作的社区维护版本，不是 Celechron 官方发布版本。感谢原项目作者和所有贡献者。

### 本次主要更新

#### 一、修复并加固统一身份认证和刷新流程

重新梳理了统一身份认证、一次性 ticket、各服务登录态以及已认证 Session 的获取流程。

主要改进包括：

- 修复部分情况下看似登录成功，但实际仍处于匿名状态的问题；
- 登录后会进一步检查服务端认证状态；
- 登录态失效时支持重新认证；
- 为网络请求增加超时、重试和异常分类；
- 减少因为某一个数据源暂时异常，导致整个刷新过程完全失败的情况；
- 优化前台刷新和后台刷新同时执行时的冲突问题，避免重复请求和状态覆盖。

在网络不稳定或者校内系统偶发超时时，刷新过程会比原版更加稳定。

#### 二、支持主动探测下一学年课表

原版根据当前月份判断“现在属于哪个学年”，因此在暑假期间，即使下一学年的课表已经开放，程序也可能完全不发起请求。

例如，2026 年 7 月时，原逻辑仍然只查询到 2025—2026 学年，因此无法获取已经发布的 2026—2027 秋冬课表。

新版本不再等待固定月份，而是：

- 正常获取当前及历史学年的课表；
- 主动探测下一个学年；
- 如果下一学年课表已经开放，就自动读取并保存；
- 如果尚未开放、返回空数据或者 404，则静默跳过；
- 探测失败不会影响其他学年正常刷新；
- 不会请求毕业年份之后的课表。

目前已经实际验证，2026—2027 秋冬学期课表能够正常主动请求并刷新出来，不需要等到 9 月。

#### 三、实践栏目改为优先读取素质拓展平台

原版实践栏目主要读取教务网提供的实践分汇总，只能看到第二、第三、第四课堂的总分，无法查看具体项目。

新版本接入了素质拓展平台，可以显示更加完整的实践项目明细，包括：

- 项目名称；
- 获得记点；
- 第二、第三或第四课堂分类；
- 审核状态；
- 项目类别；
- 素质类别；
- 参与身份或得分原因；
- 情况说明；
- 活动开始和结束时间；
- 最近更新时间。

项目会分为：

- 已审核并计入总分；
- 审核中或暂未计入。

如果素质拓展平台临时无法访问，会依次尝试：

- 使用素质拓展平台缓存；
- 使用教务网原有的实践分汇总；
- 保留已有数据并提示当前数据来源。

因此，即使某个数据源临时不可用，也尽量不会直接显示为空。

#### 四、增加在线校历和多级回退机制

校历配置改为按学期在线获取，使后续校历调整不必完全依赖重新发布 App。

校历获取具有多级回退：

- 优先使用在线最新配置；
- 在线获取失败时使用该学期本地缓存；
- 没有对应缓存时，参考上一份有效配置生成安全的默认校历。

同时增加了校历数据格式检查，避免异常日期、缺失字段或尚未发布的新学期校历导致整个课表刷新失败。

程序还会标记当前数据属于：

- 实时成功；
- 使用缓存；
- 使用默认配置；
- 当前不可用。

#### 五、新增“诊断与测试”日志页面

设置页面中增加了诊断日志功能，用于排查登录、课表、成绩、考试、作业、实践和校历等数据源的问题。

支持：

- 查看最近的刷新日志；
- 复制日志；
- 导出并分享 TXT 文件；
- 清空日志；
- 查看每个数据源是实时成功、使用缓存还是刷新失败；
- 记录请求耗时、重试、重新登录和异常类型。

为了保护账号安全，导出日志时会自动隐藏：

- 学号等账号信息；
- 密码；
- Cookie；
- Session；
- ticket；
- token；
- 完整带参数的敏感链接。

日志最多保留最近 2000 行和最近 10 个导出文件。

以后遇到刷新失败，可以直接导出日志反馈，不再只能根据一句“刷新失败”猜测具体问题。

#### 六、提高接口数据解析容错能力

部分校内接口中的数字、字符串、空值和字段结构并不完全稳定。原版在遇到单条异常数据时，有时会导致整个学期甚至整个刷新过程失败。

新版本统一加强了 JSON 和字段解析：

- 兼容数字和字符串形式的字段；
- 对空值和缺失字段进行安全处理；
- 跳过无法识别的单条异常课程、成绩或考试；
- 保留其他能够正常解析的数据；
- 为异常响应补充更明确的错误信息；
- 对缓存内容同样进行格式检查。

目标是尽可能做到“一条异常数据不拖垮整个刷新”。
