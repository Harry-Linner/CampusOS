# CampusOS 插件市场合规风险分析

**Date:** 2026-06-18
**Tier:** T3 · S3
**Scope:** 假设未来引入 Mod 市场（V2）并进行商业化运营时的合规评估（当前路线未采用）
**Related docs:** [PRD](..\PRD.md) · [plan](..\plan.md) · [research](..\research.md)

> ⚠️ **免责声明：** 本文档基于公开法律法规的检索分析，不构成正式法律意见。在作出任何商业化决策前，建议咨询持证律师。

> **状态说明（2026-07-12）：** CampusOS 当前路线是完全开源、非商业、无收费、无插件市场。本文保留为"如果未来策略变化"时的假设分支，不属于当前 roadmap。

---

## TL;DR

- 当前路线（纯本地、无后端、无收费、无市场）**几乎零合规风险**
- V2 Mod 市场的**最大法律风险是"二清"（无证经营支付业务）——可入刑**
- ICP 许可证是 V2 上线前的硬门槛，缺证可罚 ¥10–100 万
- 税务主体必须合规：建议至少注册个体工商户，最佳方案是有限责任公司
- MIT 开源协议合规本身简单（保留版权声明即可），但需警惕第三方插件混入 GPL 代码的传染风险
- 2025 年三项新规显著提高了平台经营者合规要求：支付新规 + 收费透明指南 + 网络交易管理办法修正；但在当前非商业路线下，这些要求暂不触发

## 2. 分项详细分析

### 2.1 "二清"风险 — 🔴 最致命

**问题：** Mod 市场收费的典型流程是"用户付费 → 平台收款 → 扣掉抽成 → 余款打给插件开发者"。这个流程中，如果平台把用户的钱先收到**自己的银行/支付宝/微信账户**，再转给开发者，就构成了"资金二清"。

**法律依据：**
- 国务院《非银行支付机构监督管理条例》(2024.05.01 施行)：未经中国人民银行批准，任何单位和个人不得从事或变相从事支付业务
- 中国人民银行《关于进一步加强无证经营支付业务整治工作的通知》：以"大商户"模式接入持证机构，留存商户结算资金并自行清算 = 无证经营
- 《刑法》第 225 条：非法经营罪，情节严重的处 5 年以下有期徒刑或拘役，并处违法所得 1–5 倍罚金

**合规路径（必须做到）：**

```
❌ 违规模式:
用户付款 → 平台账户（资金归集） → 平台手动分账 → 开发者

✅ 合规模式:
用户付款 → 持牌支付机构（支付宝/微信支付）
                    ↓ 实时分账
         ┌─────────┴──────────┐
    开发者账户（货款-抽成）   平台账户（仅抽成/技术服务费）
```

核心原则：**平台全程不碰钱。** 资金由持牌支付机构（支付宝分账产品、微信支付分账）直接清算到开发者和平台各自账户。平台只收自己那部分抽成。

**实操建议：**
- 接入支付宝"分账"产品或微信支付"服务商分账"能力
- 每个插件开发者在支付机构完成"商户进件"，拥有独立的商户号
- 交易时即确定分账比例（开发者 85% + 平台 15%），实时分账
- 平台账户只收抽成，绝不归集交易资金

**规模阈值：** 即使年交易额只有几万块钱，"二清"的定性不以金额为转移——归集 + 清算 = 违规。但执法优先级确实与规模相关：小微平台被央行主动查处的概率较低，但一旦被举报或卷入纠纷，合规问题会被翻出来。

### 2.2 ICP 许可证 — 🔴 硬门槛

**问题：** 运营一个以盈利为目的的在线平台（Mod 市场 = 信息发布 + 交易撮合 + 收费），需要** ICP 许可证（B25 类增值电信业务许可证）**，而不只是免费的 ICP 备案。

**ICP 备案 vs ICP 许可证：**

| | ICP 备案 | ICP 许可证 |
|---|---|---|
| 性质 | 行政登记 | 行政许可 |
| 费用 | 免费 | 申请费 + 代理费 |
| 适用 | 所有境内服务器的网站/App | **经营性的**信息服务平台 |
| 违规后果 | 关闭网站 | 没收违法所得 + 罚款 ¥10 万–100 万 + 关闭 |

**CampusOS Mod 市场为什么需要 ICP 许可证：**
- 提供插件上传、展示、下载、交易撮合 = "信息发布平台"
- 收取交易抽成 = 经营性
- 归类为 B25 类"信息服务业务"

**ICP 许可证申请条件：**
| 条件 | 具体要求 |
|------|----------|
| 注册资金 | ≥ ¥100 万（认缴即可） |
| 社保人员 | 至少 3 名员工近 1 个月社保记录 |
| 经营范围 | 营业执照须含"增值电信业务" |
| 域名 | 已完成 ICP 备案 |
| 服务器 | 托管于国内持 IDC/ISP 资质的服务商 |
| 外资限制 | 外资持股 ≤ 49%（非试点区域） |

**实操建议：**
- V1 不需要（本地 App，无后端）
- V2 市场上线前，必须完成 ICP 许可证申请（审批周期法定 60 工作日，实际 2–4 个月）
- 注册资金 ¥100 万是认缴制，不一定需要实缴——但部分省份实际审核时要求部分实缴
- 建议注册在海南/上海/北京/深圳（增值电信业务外资准入试点），未来融资更灵活

### 2.3 税务合规 — 🟠 高

**问题：** 平台收入（抽成 + 官方插件付费）需要合法纳税主体。以个人身份长期经营有被税务机关认定"未办理税务登记"的风险。

**主体选择对比：**

| 主体 | 增值税 | 所得税 | 优缺点 |
|------|--------|--------|--------|
| **个体户** | 小规模 1%（季度 ≤ ¥30 万免税） | 经营所得 5%–35% 累进 | 注册简单、税负低；无限责任；融资困难 |
| **有限公司** | 小规模 1% / 一般纳税人 6% | 企税 5%（小微） + 分红 20% | 有限责任、可融资；注册复杂、维护成本高 |

**实操建议（分阶段）：**

- **MVP–V1 阶段（无收入）：** 可以不注册主体，纯粹开源项目
- **V2 市场开始收费时：** 至少注册**个体工商户**。季度收入 ≤ ¥30 万时增值税全免，年利润 ≤ ¥3 万时所得税仅 5%
- **年收入超过 ¥50 万时：** 升级为**有限责任公司**。小微企业所得税仅 5%（利润 ≤ ¥300 万），有限责任保护个人资产
- **金税四期已全面运行：** 银行流水、支付宝/微信商户号收入全部联网——不要试图用个人账户隐匿收入

**软件产品增值税优惠：** 符合条件的自行开发软件产品，增值税实际税负 > 3% 的部分可即征即退（财税〔2011〕100 号）。官方自研插件如果认定为软件产品销售（13% 税率），可以享受此优惠。

### 2.4 平台收费合规 — 🟡 中

**问题：** 2025 年 7 月 31 日，国家市场监管总局发布《网络交易平台收费行为合规指南》（2025 年第 32 号公告），适用于所有收取佣金/抽成/服务费的平台经营者，包括插件市场。

**核心要求：**
1. **公示义务** — 在首页显著位置持续公示所有收费项目、规则、标准
2. **明细透明** — 向插件开发者提供完整收费明细
3. **修改规则** — 变更收费规则须公示 ≥ 7 日并公开征求意见；开发者可选择退出
4. **历史存档** — 保存修改生效前**三年**的全部历史版本
5. **八项禁止行为：** 不得重复收费、不得只收费不服务、不得转嫁自身成本、不得向开发者收取基础经营数据费用、不得强制推广收费、不得用不合理保证金变相收费、不得价格歧视、不得收取其他不合理费用

**实操建议：**
- V2 市场开发时就把收费规则的公示页面纳入设计
- 抽成比例保持简单透明（如统一 15%），不做复杂分级
- 所有收费规则变更走 7 日公示流程
- 不做保证金 — CampusOS 不是淘宝，插件开发者不需要交保证金

---

### 2.5 插件内容审核责任 — 🟡 中

**问题：** 平台对插件内容有审核义务。如果插件含有违法违规内容（恶意代码、数据窃取、色情/赌博/政治违规内容），平台可能承担连带责任。

**法律依据：**
- 《网络安全法》第 47 条：平台应当加强对用户发布信息的管理
- 《网络交易监督管理办法》(2025 修正)：平台经营者应建立检查监控制度
- "避风港原则"仍然适用：接到侵权通知后及时删除可免责，但"明知或应知"的不适用

**实操建议：**
- V2 市场建立插件上传审核流程（人工 + 自动化静态扫描）
- 插件 manifest 与源代码均需审核
- 建立投诉/举报 + 下架机制
- 在开发者协议中明确内容合规责任和侵权赔偿条款
- 购买平台责任险？（在大规模运营后考虑）

---

### 2.6 应用程序备案 — 🟡 中

**问题：** 2023 年工信部启动 App 备案制度，2025 年起浏览器插件也已纳入备案。桌面应用的备案要求目前处于**灰色地带**——法律没有明文要求桌面软件的独立备案，但如果桌面应用包含"信息服务功能"（如在线市场），可能被要求备案。

**实操建议：**
- CampusOS 桌面端 App 本身 → 仅提供本地功能，暂不需要备案
- Mod 市场网页端（如果有）→ 必须完成 ICP 备案，市场功能还需要 ICP 许可证（见 2.2）
- Mod 市场 API（市场后端服务）→ 需要 ICP 备案
- 持续关注工信部是否将桌面应用纳入备案范围

### 2.7 个人信息保护 — 🟡 中（V2）

**问题：** 如果 V2 引入云同步和用户账户系统，将涉及用户个人信息的收集和处理，触发《个人信息保护法》(PIPL, 2021)。

**核心要求：**
- 收集前告知 + 取得同意
- 最小必要原则
- 用户有权查询、更正、删除、导出
- 数据泄露需 72 小时内报告
- 向境外传输需安全评估或认证

**V1 阶段：** 纯本地存储，不收集任何用户数据 → PIPL 不适用（不构成"处理"）

**实操建议：**
- 统一认证密码只允许在用户主动点击“连接并保存”后经 context-isolated IPC 进入主进程；验证失败不得持久化，Cookie、Session 和 ticket 不得进入 renderer、插件、日志或磁盘缓存
- Windows 凭据当前由 Electron `safeStorage`/DPAPI 保护；账号、验证时间及 `getMyInfo` 的二/三/四课堂汇总回执仍属于个人信息，只允许本地最小必要处理。文件权限、删除入口和后续导出功能都应覆盖这些字段；Cookie、Session、ticket 和原始响应不得落盘
- 自有账号现场验收只允许通过进程环境变量注入；脱敏测试输出仅包含阶段布尔结果，不得打印学号、密码、素拓汇总、Cookie、Session、ticket、token、响应正文或完整敏感 URL
- 校内数据接入的诊断日志默认仅保存在本机；复制或导出前必须脱敏账号、密码、Cookie、Session、ticket、token 和敏感 URL 参数，分享动作只能由用户主动触发
- 如果未来增加自动上传诊断、遥测或远程排障，必须重新评估个人信息处理边界，并在上传前提供明确告知、最小范围选择和单独同意
- V2 云同步设计时遵循"最小收集"原则
- 提供清晰的隐私政策（不是复制粘贴模板）
- 端到端加密传输（已在 Lisa 规格中规划）
- 服务器放在国内（阿里云/腾讯云），避免跨境数据传输

### 2.8 MIT 协议合规 — 🟢 低

**问题：** CampusOS 核心框架 MIT 开源，Mod 市场闭源。MIT 是**最宽松的开源协议**，允许闭源商用——唯一的红线是**必须保留原始版权声明和许可文本**。

**已知踩坑案例：**
- **CEC-IDE 事件**（国内标志性）：某公司拿 VSCode（MIT）改头换面自称"自建插件市场"，删除了 MIT 版权声明 → 开发者社区声讨 → 被迫道歉
- **何同学事件**（2024）：千万粉 B 站 UP 主使用 MIT 协议开源项目但删除原作者署名 → 广泛批评

**实操建议：**
- 在 CampusOS 的"关于"页面中保留 MIT License 全文和原始版权声明
- 插件 SDK 和 API 文档同样 MIT 开源，附带版权声明
- 如果有人 fork CampusOS 做闭源衍生品 → 合法（MIT 允许），但必须保留版权声明

---

### 2.9 第三方插件 GPL 传染 — 🟢 低（可规避）

**问题：** 如果社区插件开发者使用了 GPL 协议的代码，并且将插件打包为 `.campusmod` 分发，GPL 的"传染性"可能要求该插件的全部代码以 GPL 开源。但这只影响该插件本身，**不会传染到 CampusOS 核心框架或 Mod 市场**——因为插件通过 API 与主程序通信，属于"独立程序"而非"衍生作品"。

**实操建议：**
- 在插件开发文档中明确说明协议选择的影响
- Mod 市场审核时检查插件的依赖许可证
- 核心框架的插件 API 保持"进程边界"，确保法律上的独立性
- 市场前端展示插件的开源协议标签
- Celechron 1.3.0 仅作为校内数据接入的行为与架构参考；被 Git 忽略的本地参考副本不得参与 CampusOS 构建、打包或发布，也不得直接复制、改写或移植到 MIT 核心
- Celechron 启发的官方插件集必须保持 clean-room 实现证据：需求与协议说明可引用行为结论，代码评审不得以 Celechron 源文件为补丁来源；构建和许可证扫描需覆盖每个官方连接器与功能插件
- 2026-07-19 的本科教务与学在浙大连接器为独立 TypeScript 实现；核心只向已授权官方连接器返回固定课表/考试/作业操作的业务正文，不暴露密码、Cookie、Session、ticket、请求头或通用网络句柄。课表、考试和作业正文属于本地个人学业数据，必须继续受权限、provenance、删除入口和诊断脱敏约束；官方校历连接器只读取浙江大学公开 HTTPS 页面，不携带个人会话。当前未移植 Celechron 源文件
- 如需复用任何 Celechron GPL-3.0 源码，必须先通过 ADR 明确派生作品边界、开源发布义务和第三方声明，再决定是否采用

| 阶段 | 收入模式 | 最低合规要求 | 建议主体 | 预估合规成本 |
|------|----------|-------------|----------|-------------|
| **MVP (Phase 1–3)** | ¥0, 开源免费 | 无 | 个人 | ¥0 |
| **V1 (Phase 4)** | ¥0, 靠口碑增长 | 无 | 个人 | ¥0 |
| **V2 市场启动** | 平台抽成 + 官方插件付费 | ICP 许可证 + 个体户/公司 + 合规分账 | 有限公司 | ¥5k–15k (ICP 代理 + 注册 + 会计) |
| **V2 市场运营** | 同上 + 可能融资 | 全部上述 + 隐私政策 + 内容审核 | 有限公司 | ¥2k–5k/月 (会计 + 合规维护) |
| **规模化 (多校覆盖)** | 同上 + 可能 B2B | 全部上述 + 等保测评 + 法律顾问 | 有限公司 | ¥10k–30k/月 |

## 4. 关键判断

1. **MVP 阶段完全不需要担心合规 — 本地 App、无后端、无市场、无收入。** 先把产品做出来，把用户验证做了。不要因为还没发生的合规问题而不开始。

2. **V2 市场的最大风险是"二清"——这不是罚款的问题，是可入刑的。** 而且解决"二清"的关键不是申请支付牌照（中小企业根本拿不到），而是**设计资金流时让持牌机构做清算，平台不碰钱**。这是架构问题，不是法务问题。

3. **ICP 许可证是 V2 市场的硬门槛，申请需要 2–4 个月。** 在 V1 验证了产品价值、决定做 V2 时，ICP 许可证应该是排在第一位的待办事项——在写任何市场后端代码之前。

4. **2025 年的三项新规（支付新规 + 收费透明指南 + 网络交易管理办法修正）说明监管在加速收紧。** "先做大规模、再补合规"的策略在 2026 年已经不适用了。但 CampusOS 的优势是：它从 V1 的"纯本地、无后端"起步，天然避开了大部分监管触发点。V2 市场是收入模式升级，也是合规责任升级。

5. **如果 CampusOS 继续保持当前路线：不上 Mod 市场、不收费、优先纯本地**——它就是一个纯桌面端开源工具，合规风险几乎为零。

---

## 5. 一句话总结

> **用一句话回答：如果未来真的做收费插件市场，合规风险很高，而且有一个致命的（二清）和一个硬门槛的（ICP 许可证）；但在 CampusOS 当前完全开源、非商业的路线下，这些都不是眼前的问题。**
